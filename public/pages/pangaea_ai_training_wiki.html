● COURSE 1
The Basics

Everything you need to use AI chatbots effectively — from what they actually are, to how to ask good questions and avoid the most common pitfalls.

1 · What Is an LLM?

A Large Language Model (LLM) is the core system behind modern AI tools, trained on large collections of text — such as books, websites, and scientific literature — to learn patterns in language. Its fundamental mechanism is predicting the next token (a small unit of text) based on the given context, with capabilities shaped further by instruction tuning and human feedback. As a result, LLMs can perform tasks such as summarising documents, generating code, and explaining complex concepts.

CORE INSIGHT

An LLM does not "know" facts or understand information in a human sense; it generates outputs based on statistical patterns rather than verified knowledge. Its responses should therefore be interpreted as plausible constructions, not guaranteed truths, and can include confident but incorrect statements.

Performance is sensitive to how prompts are phrased and tends to degrade in tasks requiring precise accuracy, complex multi-step reasoning, or highly specialised, up-to-date domain knowledge.

WHAT LLMs ARE GOOD AT...
→ Summarising and explaining text
→ Drafting, editing, translating
→ Writing and checking code
→ Answering factual questions (common knowledge)
→ Spotting patterns in structured text
→ Processing large volumes quickly
WHAT LLMs STRUGGLE WITH...
→ Precise arithmetic (without tools)
→ Obscure or very recent facts
→ Knowing when they're wrong
→ Accessing real-time data unless connected to search
→ Novel, poorly specified, or long-horizon problems
PANGAEA APPLICATION

LLMs can help curators draft parameter descriptions, suggest unit annotations, check that metadata text is clear, or explain unfamiliar scientific terms. They should not be trusted to generate precise coordinate values or date/time entries without human verification.

💡 HALLUCINATION

When an LLM confidently states something false. This happens more often for obscure topics, specific numbers, or citations. Always verify important facts independently.

Domain-Specific Failure Modes (Earth & Environmental Data)

In scientific data curation, AI errors often appear plausible but introduce subtle inconsistencies. Most errors are not random — they are systematic and plausible, making them harder to detect.

Failure ModeExamples
Unit confusion°C vs K, mg/L vs µmol/L — LLMs do not track units as formal objects
Coordinate errorsLatitude/longitude swapped, incorrect CRS (e.g. WGS84 vs local projection)
Time handlingUTC vs local time, ambiguous date formats (e.g. 03/04/2020)
Parameter misclassificationSimilar but distinct variables (e.g. salinity vs conductivity)
False standardisationAI "normalises" terms incorrectly to familiar patterns

2 · What Is a Chatbot?

A chatbot is the interface you use to talk to an LLM. Think of the LLM as the brain and the chatbot as the mouthpiece — you interact with the mouth, but the brain is doing the work underneath.

Chatbots accept your text (and sometimes images or files) and return a response. Most modern chatbots maintain the conversation in memory during a session, so you can follow up, correct, and refine across multiple messages.

HOW A SINGLE MESSAGE WORKS...
→ You type a message (the prompt)
→ The chatbot adds context and system instructions
→ Everything goes to the LLM as one combined input
→ The LLM generates a response token by token
→ The response streams back to your screen
KEY CONCEPTS...
→ Prompt — your input message
→ Context window — max text the model can hold
→ Token — unit text is measured in (~¾ of a word)
→ System prompt — hidden instructions set by the app
→ Temperature — controls output variability

Temperature

Temperature is a numerical parameter (typically 0.0–1.0) that controls how the model selects the next token at each step of generation.

At low temperature (0.0–0.3), the model strongly favours the highest-probability token at each step. Output is more consistent, predictable, and repeatable. This is preferred for factual tasks, coding, and structured outputs where correctness matters more than variety.

At higher temperature (0.7–1.0), the model samples more broadly across probable tokens. Output becomes more varied and less predictable. This is useful for brainstorming, exploring alternative phrasings, or creative tasks.

Important: Temperature is set by the platform or via the API — it is not a setting you control in a standard chatbot interface. You cannot reliably set temperature through natural-language instructions like "be more creative." However, the way you phrase your prompt does influence output variability indirectly: open-ended prompts ("suggest several approaches") tend to produce more diverse responses than tightly constrained ones ("give me the single correct unit"), regardless of the temperature setting.

PANGAEA APPLICATION

In curation work, chatbots can act as a first pass for writing — drafting dataset descriptions, suggesting parameter labels, or explaining incoming data formats. The curator remains responsible for the final output.

4 · Data Privacy & Responsible Use

AI chatbots are powerful tools, but every message you send leaves your device and is processed on external servers. For scientific data curators handling unpublished research data, this creates real obligations.

What You Must Not Share

Treat every chatbot prompt as if it were a public post. Do not paste: unpublished dataset content or raw data files, personally identifiable information (researcher names, emails, ORCID IDs in combination with unpublished work), confidential institutional documents or internal communications, login credentials, API keys, or access tokens.

Free vs. Paid Tiers — What Changes

Free tiers of most chatbots may use your conversations as training data. This means your input could, in principle, influence future model outputs. Paid tiers and API access typically offer opt-out from training data use, but the specifics vary by provider and change over time. Always check the current terms of service.

Practical Rules for Curators

1. Use anonymised or synthetic examples when testing prompts with real data structures.

2. Strip identifying metadata (researcher names, campaign-internal codes) before pasting into any chatbot.

3. For routine curation tasks (drafting descriptions, explaining methods), generic context is usually sufficient — you rarely need to share actual data.

4. When in doubt, ask: "Would I be comfortable if this text appeared in a public dataset?" If not, do not paste it.

KEY PRINCIPLE

AI tools are external services, not internal infrastructure. Apply the same data hygiene you would use when emailing a colleague outside your organisation.

Before asking an AI to help annotate a dataset, run through 4-D: Delegate (what exactly am I asking it to do?), Describe (give it parameter context and examples), Discern (check every suggested annotation against controlled vocabulary), Diligence (you as curator sign off on the final metadata).

6 · How to Use Chatbots Effectively

Chatbots are not search engines. The quality of what you get back depends almost entirely on the quality of what you put in. Think of it as working with a very capable but literal-minded colleague who only knows what you tell them.

1. Let the AI help you build a better prompt first — Before diving into your task, describe what you want to achieve and ask: "I want to [goal]. Help me write a clear, effective prompt I can use for this." Review and adjust its suggestion.

2. Start with context — Tell the chatbot who you are and what you need. "I'm a data curator at a scientific database. I need help writing a description for a temperature dataset" gives much better results than just "write a description."

3. Be specific about the output — State the format, length, tone, and audience. "Write 2–3 sentences in plain English for a metadata field" prevents a two-page essay.

4. Treat the first response as a draft — Follow up: "Make it shorter," "add a note about the measurement method," "use more formal language." This is normal and expected.

5. Check everything important — Verify facts, numbers, and citations independently. Chatbots sound confident even when wrong.

6. Ask for alternatives — "What are two alternative ways to phrase this?" or "What are the weaknesses of this description?" Comparing options improves results.

💡 TIP

The AI never asks clarifying questions unless you tell it to. If your request is ambiguous, the chatbot will make assumptions and proceed — often confidently in the wrong direction. Be explicit.

"I Corrected the Chatbot — Now It Will Always Give the Correct Answer"

This is a common misconception. When you correct a chatbot, your correction only applies to the current conversation. The model adapts its responses within the active context window, but no permanent learning takes place from a single interaction.

Behind the scenes, providers collect feedback (thumbs up/down, flagged responses, selected conversation logs) and aggregate it over time. This data may eventually feed into evaluation datasets, supervised fine-tuning, or reinforcement learning — but only in future model versions, not in real time.

Real-time learning is deliberately prevented to avoid manipulation and data poisoning, and to maintain consistent, reproducible model behaviour. Some platforms offer personal memory features that store user-specific preferences across sessions, but these are local workarounds — not global model changes.

PRACTICAL TAKEAWAY

Your correction improves the current chat. Reporting a problem contributes to long-term system improvement. Neither changes the model for other users in real time.

7 · Prompting — How to Ask Well

Prompting is the skill of structuring your input so the AI produces the output you actually want. Eight techniques, in order of impact:

Note: with model updates, prompt outputs can change!

01 · Give Context

Background shapes every word of the response. Tell the AI your role, the purpose, and any relevant constraints.

I'm a data curator at PANGAEA, preparing a parameter description for publication in a scientific database.

02 · Show Examples

One good example is worth a paragraph of description. Show the AI what good looks like.

Here is an existing good description: [paste]. Write a similar one for sea surface salinity.

03 · Specify Output Format

Define length, structure, language, and audience. Never assume the AI will guess correctly.

Max 2 sentences. Scientific English. For a metadata field visible to researchers in the database.

04 · Assign a Role

Telling the AI to adopt a specific persona dramatically shapes its tone and vocabulary.

You are an expert in oceanographic data curation. Review this parameter annotation and flag any issues.

05 · Break into Steps

Complex tasks work better when split. Verify each step before moving to the next.

Step 1: Identify the measurement type. Step 2: Check the unit. Step 3: Write the description.

06 · Give Space to Think

For hard problems, ask the AI to reason before answering. Reduces confident-but-wrong responses.

Before answering, consider the possible unit conventions used in this discipline, then respond.

07 · Chain-of-Thought

For complex tasks, request structured intermediate outputs rather than always exposing full reasoning.

Think step by step. Show your reasoning at each stage before giving the final answer.

08 · Iterate Relentlessly

Treat first responses as drafts. The best outputs come from dialogue, not single prompts.

Good, but make it more concise. Remove the third sentence and add the measurement method.
PANGAEA APPLICATION

Practical Examples for Curators

→ Formulate emails (with appropriate tone)
→ Check naming, typos, and abbreviations (works well for chemical elements)
→ Explain methods — what is the model, the type, what does it do?
→ Explain parameter/unit — what is it, does the unit make sense?
→ Rephrase text or abstracts
→ Write code snippets for Jupyter Notebooks

Common Prompting Mistakes (Anti-Patterns)
Anti-PatternProblem

KEY PRINCIPLE

If the task is unclear to a human colleague, it is unclear to the AI.

8 · Context Rot — Why Long Conversations Degrade

Every AI conversation happens inside a context window — a fixed-size memory buffer. The model doesn't remember between sessions; instead, it re-reads the entire conversation on every reply. When that window fills up, something predictable and problematic happens.

WHEN DOES TEXT FALL OUT OF THE CONTEXT WINDOW?

Context windows are measured in tokens (~¾ of a word each). Modern chatbots handle large windows, but long sessions fill them faster than you'd expect:

→ Claude (claude.ai): ~200,000 token window (up to 1M depending on model). Quality degradation typically sets in well before the hard limit, around 50–100 long exchanges.

→ ChatGPT and Gemini have similarly large windows (100k–1M tokens). In practice, context rot sets in before the hard limit — it's a gradient, not a cliff.

System prompt vs. your first message: The system prompt is a hidden block of instructions set by the platform — pinned and protected, it never scrolls out. Your first message is a normal user message and can scroll out in long conversations. Re-state important instructions near the end of long sessions, or start a fresh chat.

SYMPTOMS OF CONTEXT ROT...
→ Contradicts something agreed earlier
→ Style instructions silently stop being followed
→ Corrected mistakes reappear
→ Drifts toward immediate question, away from original goal
→ Output quality subtly degrades
WHAT TO DO ABOUT IT...
→ Start fresh — new chat for a new task
→ Distil before continuing — short summary, not whole conversation
→ Anchor key constraints — re-state important rules near current message
→ Chunk long tasks — split into self-contained stages
→ Spot the symptoms — treat subtle drift as a signal
KEY PRINCIPLE

Context rot is not a bug to be patched. It is a structural consequence of how attention-based AI models work. 200 words of clean, distilled context beats 3,000 words of wandering dialogue every time.

9 · Trust Boundaries — What to Trust, What to Verify

AI systems are highly capable, but not uniformly reliable across all types of tasks. In data curation, it is critical to distinguish between outputs that can be used directly and those that require strict verification.

LOW RISK — Generally Safe

Rewriting text for clarity · Summarising documents · Translating between languages · Suggesting alternative phrasings · Formatting and structuring text

MEDIUM RISK — Requires Verification

Explanations of scientific concepts · Unit interpretations · Parameter descriptions · Code suggestions

HIGH RISK — Must Always Be Verified

Numerical values (coordinates, measurements, timestamps) · Units and unit conversions · Citations and references · Data transformations · Classification into controlled vocabularies

KEY PRINCIPLE

The closer an output is to publication-critical data, the less it should be trusted without validation. Unit conversions should be handled by code, not language models.

1 · Coding Agents

A coding agent is an AI that doesn't just chat — it takes actions. It can read and write files, run code, call external services, and loop through multiple steps autonomously until a task is complete. The most prominent example is Claude Code, a command-line tool released by Anthropic in early 2025.

THE KEY DIFFERENCE FROM A CHATBOT

A chatbot waits for you to approve each step. A coding agent is given a goal and executes a sequence of actions — file operations, code runs, web fetches — largely autonomously. Agents can execute multi-step actions, often with optional or configurable human oversight.

WHAT CODING AGENTS CAN DO...
→ Write and run entire Python scripts
→ Process batches of files automatically
→ Search the web and incorporate results
→ Create and update documents, spreadsheets, presentations
→ Connect to external services via MCP
→ Loop and self-correct until a task is done
NEW RISKS VS. CHATBOTS...
→ Irreversible actions — can delete files, send emails
→ Prompt injection — malicious text can redirect the agent
→ Scope creep — agents sometimes solve adjacent problems too
→ Compounding errors — early mistakes propagate through all steps
PANGAEA APPLICATION

Example: pyPanTool — a batch file processing mini-desktop application (in development) created with Claude Code. It automates repetitive file operations (format conversion, quality checks, batch renaming) for large files (>600 MB). As a non-programmer, I cannot write Python code myself, but I can now use and adapt my own tool.

2 · Compounding Error in Sequential AI Tasks

Every time an AI takes an action, there is some probability it gets it slightly wrong. In a single chatbot exchange, this is manageable — you check the output. But in a long chain of automated steps, small errors compound mathematically.

PROBABILITY OF A FULLY CORRECT OUTPUT = accuracyn

n = number of sequential steps

99% accuracy × 10 steps → 90.4% (still high — manageable)
99% accuracy × 50 steps → 60.5% (a coin-flip on correctness)
99% accuracy × 100 steps → 36.6% (likely wrong)
95% accuracy × 100 steps → 0.6% (near-certain failure)
99% accuracy × 1,000 steps → 0.004% (essentially zero)

WHY THIS IS WORSE IN PRACTICE...
→ Errors are often systematic, not random
→ Early mistakes change how later steps are interpreted
→ The system builds on its own incorrect assumptions
→ Outputs remain plausible, making detection difficult
DESIGN RESPONSES...
→ Break workflows into independent, verifiable stages
→ Insert human checkpoints at high-impact steps
→ Validate critical fields with code, not LLMs
→ Use structured outputs to make inconsistencies detectable
→ Re-run critical steps with alternative prompts
BOTTOM LINE

This is not a flaw to be fixed — it is a mathematical property of sequential processes. Any workflow design for agentic AI must account for it. Short, verifiable chains with human checkpoints outperform long, fully automated ones.

3 · Human-in-the-Loop — When and How

AI systems are most effective when combined with structured human oversight. The right tool and the right level of human involvement depend on whether a task is exploratory (you're figuring things out) or operational (you know what needs to happen and want it automated).

Choosing the Right Tool

Situation Best Tool Why
Draft a parameter descriptionChatbotExploratory, iterative, needs your judgment
Explain an unfamiliar term or methodChatbotSingle question, conversational
Check 500 metadata records for format issuesCoding agentHigh volume, rule-based — agents excel
Convert 200 .dat files to tab-delimitedCoding agentBatch processing, defined rule
Decide if a dataset meets quality criteriaHuman + chatbotRequires judgment; chatbot assists, not decides
Match parameters to controlled vocabularyAgent + human reviewAI suggests; curator approves before publication
⚠ RULE OF THUMB

If a mistake in one step would propagate to 50 other records, build a human checkpoint before that step. Automation multiplies both speed and errors equally.

Common Human-in-the-Loop Patterns

Draft → Review → Publish — AI generates a first version (e.g. parameter description). Curator reviews, edits, and approves.

Suggest → Approve → Apply — AI proposes classifications or mappings. Curator validates before changes are applied to the database.

Batch Process → Sample Check → Release — AI processes large datasets. Curator validates a subset (e.g. 5–10%). Release only if error rate is acceptable.

Assist → Decide — AI provides analysis or options. Final decision remains with the curator.

Why Structure Matters

AI increases speed, but also scales errors. Human checkpoints ensure that mistakes do not propagate across datasets. Structured workflows break tasks into stages (planning, execution, validation) and introduce checkpoints — similar to project management or data processing pipelines. This makes intermediate results visible, testable, and correctable before proceeding.

KEY PRINCIPLE

Fully automated workflows are rarely appropriate for scientific data curation. The question is never "chatbot or agent?" but "how much human oversight does this task need?"

4 · RAG · MCP · Skills — Advanced AI Architecture

Three concepts that significantly extend what AI tools can do — moving from a chatbot you type at, toward a system that connects to your data and acts on your behalf.

📚 RAG — RETRIEVAL-AUGMENTED GENERATION

Gives the AI access to your documents. Before generating a response, the system searches a knowledge base for relevant chunks and injects them into the prompt.

Key property: Passive — content is pre-retrieved before the AI thinks. RAG reduces hallucination risk but does not eliminate it; errors can arise from retrieval and context selection.

🔌 MCP — MODEL CONTEXT PROTOCOL

Gives the AI the ability to act — calling external tools in real time. MCP is an open protocol, originally proposed by Anthropic, that allows AI systems to connect to external tools — calendars, databases, APIs — without custom wiring for each pair. It is gaining adoption across AI platforms but is not yet a universal standard.

Key property: Active — the AI decides when and what to trigger, mid-conversation. Adds genuine agency.

🧩 SKILLS — PACKAGED REASONING MODULES

Gives the AI specialist behaviour on demand. A skill is a markdown instruction file (e.g. SKILL.md) containing best practices and step-by-step guidance for a specific type of task. When the AI recognises a relevant task, it reads the file before proceeding — encoding expert knowledge you would otherwise re-explain every time. Currently a Claude-specific convention; other AI systems implement similar ideas differently.

Key property: Reusable — write the expert instructions once, apply them consistently across many tasks.

📋 PROJECT CONTEXT FILES — AI ONBOARDING DOCUMENTS

Gives the AI persistent background knowledge about a specific project or codebase. These are plain-text files placed in a project folder that the AI reads at the start of a session — covering conventions, architecture decisions, and preferences. Unlike Skills, which encode how to perform a task, project context files encode how this particular project works. This pattern is converging across AI coding tools under different names: CLAUDE.md (Claude / Claude Code), AGENTS.md (OpenAI Codex CLI), .cursorrules (Cursor), .windsurfrules (Windsurf), and .github/copilot-instructions.md (GitHub Copilot). The concept is essentially the same across all of them — a briefing document for the AI agent, written once and reused across sessions.

Key property: Persistent context — reduces repetition and keeps AI behaviour consistent within a project.

PANGAEA APPLICATION

RAG at PANGAEA — a concrete scenario: Build a knowledge base by ingesting PANGAEA's parameter vocabulary (~40,000 entries), the curation guidelines wiki, and BSRN station documentation. Documents are chunked into semantic units (e.g. one parameter definition per chunk, one guideline section per chunk) and embedded using a sentence-transformer model.

How a curator would use it: A submitter provides a dataset measuring "SW_down" in "W m⁻²". The curator asks the RAG system: "What is the standard PANGAEA parameter for downward shortwave radiation?" The system retrieves the matching vocabulary entry and returns both the canonical parameter name and its standard unit — grounded in PANGAEA's own controlled vocabulary, not the LLM's general training data.

MCP: Connect the AI directly to the editorial system or parameter database so it can suggest and retrieve entries in real time, without copy-pasting between tools.

Skills: Encode the curation checklist as a SKILL.md file — every new AI-assisted session starts with the same expert context, enforcing consistent quality standards.

Practical Considerations for RAG Systems

RAG systems depend heavily on how documents are prepared and retrieved. Poor implementation leads to misleading but confident answers.

Design FactorDetails
Chunking strategyDocuments split into smaller pieces — too large = irrelevant context, too small = loss of meaning
Retrieval qualityMust retrieve the right chunks, not just similar ones. Semantic similarity ≠ correctness
Metadata filteringFiltering by dataset, parameter, or domain improves precision
Version controlKnowledge bases must be updated and versioned — outdated documents = outdated answers
Grounding limitationsEven with correct documents, the model may misinterpret them
KEY PRINCIPLE

RAG improves accuracy, but does not replace validation.

5 · Evaluating AI Outputs — Benchmarking and Validation

To use AI reliably in data curation, its performance must be evaluated systematically, not anecdotally. AI outputs vary between runs and model versions. Without measurement, quality cannot be controlled or improved.

Approach How
Gold standard datasetsCreate a small set of manually verified examples. Test AI outputs against them consistently.
Task-specific evaluationCheck performance separately for parameter descriptions, unit validation, classification tasks.
Error trackingLog recurring mistakes (e.g. unit confusion). Use them to refine prompts or workflows.
Sampling strategiesFor batch tasks, review a fixed percentage (e.g. 5–10%). Estimate overall error rate.
Reproducibility checksRe-run the same prompt and compare outputs. Detect instability or drift.
KEY PRINCIPLE

AI quality should be treated like data quality — measurable, monitored, and continuously improved.

6 · Token Usage — Understanding AI Costs

Everything in AI is measured in tokens — the basic unit of text that a model processes. Roughly three-quarters of a word, or about four characters. Both what you send (input) and what you receive back (output) count against your token usage and cost.

WHY TOKENS MATTER

Token limits define how much context you can use in one session. Token costs define how much automated workflows cost to run. Pasting an entire 10,000-row dataset into a prompt is both expensive and likely to cause context rot — there's a better way for every large-data use case.

TOKEN-WASTING HABITS...
→ Pasting entire files when only a section is needed
→ Repeating full context in every message
→ Asking Opus for tasks Haiku handles equally well
→ Long, unfocused prompts with multiple questions
→ Re-running failed prompts without changing the approach
TOKEN-EFFICIENT HABITS...
→ Send only the relevant excerpt
→ Use a fresh chat per task
→ Match model tier to task complexity
→ Use RAG to inject only what's needed
→ Batch similar tasks in one well-structured prompt
💡 BATCH API

For heavy batch processing (hundreds of API calls), many providers offer a Batch API at roughly half the standard cost. Tasks that don't need an immediate response — like processing 500 metadata records overnight — are ideal candidates.

7 · Structured Outputs and Schema Enforcement

For AI to be reliably integrated into data curation workflows, outputs must be predictable, consistent, and machine-readable. Free-form text is flexible but difficult to validate, compare, or reuse programmatically.

Structured outputs constrain the AI to return results in a predefined format — typically JSON, tables, or schema-based records. This enables direct integration with databases, scripts, and validation pipelines.

Why It Matters

Consistent structure allows automated quality checks. Schemas enforce required fields and valid formats. Outputs become reproducible and comparable across runs.

How to Apply

Explicitly define the output format in the prompt. Provide a schema or example structure. Validate outputs automatically where possible.

Example

Instead of a paragraph description, require a structured record with fixed fields (e.g. parameter name, unit, method, uncertainty).

KEY PRINCIPLE

Unstructured text supports exploration. Structured outputs enable reliable operations.
